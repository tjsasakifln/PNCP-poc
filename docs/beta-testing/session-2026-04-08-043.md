## Diagnóstico: O Gap de ~7.700 URLs

### Situação Anterior (Sessão 042 / início desta sessão)
- Sitemap: **2.303 URLs**
- `/cnpj/*`: apenas 55 páginas (threshold `_MIN_BIDS=3` + PostgREST limit 1k)
- `/orgaos/*`: apenas 36 páginas (threshold `_MIN_BIDS=5` + PostgREST limit 1k)
- Gap vs. SEO-ORGANIC-PLAYBOOK: **~7.700 URLs faltando**

### Root Cause Identificada (3 camadas)

**Camada 1 — Threshold muito restritivo:**
- `sitemap_cnpjs.py`: `_MIN_BIDS=3` → apenas CNPJs com ≥3 editais
- `sitemap_orgaos.py`: `_MIN_BIDS=5` → apenas órgãos com ≥5 editais
- Com 4.385+ CNPJs distintos no datalake, apenas 55 tinham ≥3 bids

**Camada 2 — PostgREST max-rows=1000:**
- A query `sb.table("pncp_raw_bids").select("orgao_cnpj").limit(50000)` retornava apenas 1.000 rows
- PostgREST impõe limite de 1.000 linhas por request por padrão
- Mesmo com `limit(50000)` no SDK, o servidor capava em 1.000

**Camada 3 — Next.js fetch cache entre builds:**
- `sitemap.ts` usava `next: { revalidate: 86400 }` nos fetch calls
- Este cache persiste entre builds de Railway, reusando dados da build anterior
- Resultado: mesmo depois de fixes no backend, o sitemap regenerava com dados velhos (55 CNPJs)

### Solução Implementada

**Fix 1:** Threshold `_MIN_BIDS = 1` em ambos os arquivos (qualquer CNPJ com ≥1 edital é válido para sitemap)

**Fix 2:** RPC SQL `get_sitemap_cnpjs_json` / `get_sitemap_orgaos_json` (RETURNS json scalar)
- Scalar JSON não é sujeito ao max-rows do PostgREST
- GROUP BY e ORDER BY executados no banco, retorna array JSON completo
- Migration: `20260408200000_sitemap_rpc_json.sql`
- Fallback paginado: se RPC não existir, loop de 1k rows/página até esgotar tudo

**Fix 3:** `cache: 'no-store'` nos fetch calls de `sitemap.ts`
- Garante que cada build do Next.js busca dados frescos do backend
- Sem dependência de cache persistente entre deployments

## Análise de Saúde do SEO — Pós-Sessão 043

### O que está funcionando ✅

1. **Sitemap 9.303 URLs:** De 2.303 → 9.303. Ondas 1+2+3 completas.
   - Onda 1: 5.001 páginas `/cnpj/*` (compradores do PNCP)
   - Onda 2: 2.001 páginas `/orgaos/*` (entidades compradoras)
   - Onda 3 (anterior): 1.215+ páginas cidade×setor
   - Base: blog + alertas + glossário + outros ~1.086

2. **Backend endpoints robustos:** RPC JSON scalar + fallback paginado garante dados completos mesmo com PostgREST max-rows.

3. **Sitemap fresco a cada build:** `cache:'no-store'` nos fetch calls garante que Railway rebuilds sempre pegam dados atualizados.

4. **GSC resubmitido:** Novo sitemap enviado, Google vai re-crawl nos próximos dias.

### O que preocupa ⚠️

1. **569 "Detectada, mas não indexada":** Dado da sessão 042 — Google detectou via sitemap mas não indexou ainda. Com +7.000 novas URLs, esse número pode crescer muito antes de estabilizar.

2. **Qualidade de conteúdo das páginas pSEO:** As 5.001 páginas `/cnpj/*` e 2.001 `/orgaos/*` precisam ter conteúdo suficiente para não serem classificadas como thin content pelo Google. Verificar na próxima sessão.

3. **ISSUE-SEO-003 reavaliação:** Com as páginas `/cnpj/{cnpj}` no sitemap (Onda 1), ISSUE-SEO-003 pode estar resolvida. Verificar se as páginas frontend `/cnpj/[cnpj]` existem e têm conteúdo.

4. **ISSUE-SEO-005 pendente:** Title do artigo atualizado, mas precisamos aguardar 2-4 semanas de dados GSC para confirmar melhora de CTR.
