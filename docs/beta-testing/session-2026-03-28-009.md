### ISSUE-025: Busca por SETOR retorna 0 resultados — Regressão Crítica

**Persona:** Priya "The Edge Lord" + Alex "The Closer"
**Severidade:** P0 — Blocker
**Area:** Buscar / Filtro / Core
**Data:** 2026-03-28 15:55 UTC
**Status:** ABERTO

#### Passos para Reproduzir
1. Login em https://smartlic.tech (admin)
2. Navegar para https://smartlic.tech/buscar
3. Selecionar modo "Setor" (default)
4. Escolher qualquer setor (Engenharia, Vestuário, etc.)
5. Selecionar qualquer UF (SP, SC, MG...)
6. Clicar "Buscar"

#### Comportamento Observado
TODOS os setores testados retornam 0 resultados:
- Vestuário + SP → "0 relevantes de 215 analisadas"
- Engenharia + SP → "0 relevantes de 423 analisadas"
- Engenharia + SC → "0 relevantes de 375 analisadas"
- Vestuário + SP (repeat) → "0 relevantes de 215 analisadas"

As licitações são encontradas nas fontes (215-423), mas NENHUMA passa pelo filtro de keywords do setor.

**Custom terms funciona parcialmente**: MG + "pavimentação" retorna 10 resultados (embora maioria irrelevante).

#### Comportamento Esperado
- Engenharia + SP deveria retornar dezenas de resultados (palavras como "obra", "reforma", "engenharia", "construção" são comuns em licitações de SP)
- Sessão 004 confirmava que Engenharia + SC retornava resultados

#### Screenshot
`docs/beta-testing/screenshots/session-009/ISSUE-025-sector-search-zero-results-SC.png`

#### Notas Tecnicas
- **Root cause provável**: Commit `c0262f8a` modificou `filter/core.py` linha ~2690: `kw_normalized = normalize_text(keyword)` antes de compilar regex. A função `normalize_text()` é aplicada ao keyword E ao texto. Mas algo na interação entre normalização e regex `\b` word boundaries pode estar impedindo matches.
- **Hipótese 1**: `normalize_text()` pode estar gerando strings vazias ou com caracteres que quebram `\b` boundaries
- **Hipótese 2**: A normalização pode estar removendo hífens/espaços de keywords compostos (ex: "construção civil" → "construcaocivil"), quebrando o match
- **Hipótese 3**: Global exclusions podem estar filtrando mais resultados do que o esperado
- **Custom terms funciona diferente**: Pode usar code path diferente que não passa pelo mesmo filtro setorial
- Teste rápido: reverter apenas a mudança `normalize_text(keyword)` em `filter/core.py` e re-testar

#### Fix Sugerido
1. **Imediato**: Reverter o fix de ISSUE-017 em `filter/core.py` (a normalização de keywords) para restaurar a funcionalidade de busca por setor
2. **Médio prazo**: Reformular o sistema de filtros — a abordagem de keywords + regex é frágil demais. Considerar "setor temporário" para custom terms (sugestão do usuário)

### ISSUE-021 (ATUALIZAÇÃO): Pipeline HTTP 500 — Regressão

**Persona:** Alex "The Closer"
**Severidade:** P0 — Blocker (UPGRADE de P1)
**Area:** Pipeline
**Data:** 2026-03-28 15:50 UTC
**Status:** REGREDIU

#### Passos para Reproduzir
1. Login em https://smartlic.tech (admin)
2. Navegar para https://smartlic.tech/pipeline
3. Observar erro

OU

1. Na busca, clicar "Pipeline" em qualquer resultado
2. Botão mostra "Erro"

#### Comportamento Observado
- GET /api/pipeline?limit=200 → HTTP 500 (múltiplas vezes)
- POST /api/pipeline → HTTP 500
- Página Pipeline: "Não foi possível carregar seu pipeline." com botão "Tentar novamente"
- Console: 3-4 erros `Failed to load resource: 500`

#### Comportamento da Sessão Anterior (008)
- GET /api/pipeline → 200 (retornava vazio)
- POST /api/pipeline → 409 (conflito)

#### Notas Técnicas
- O fix `c0262f8a` migrou pipeline routes de `get_supabase` (admin client) para `get_user_db` (user-scoped client)
- A migração provavelmente quebrou a query ou o acesso à tabela pipeline
- `get_user_db` pode não ter as permissões corretas ou pode requerer setup diferente no request context
- Network logs confirmam: primeiro GET na sessão retornou 200, depois começou a retornar 500 — pode haver race condition no client user-scoped

#### Fix Sugerido
- Reverter pipeline routes para `get_supabase` (admin client) e investigar RLS policies separadamente
- OU: verificar que `get_user_db` está recebendo o token JWT correto no request context
