### ISSUE-SEO-001: robots.txt — Google-Extended bloqueado pelo bloco Cloudflare Managed

**Persona:** Jordan
**Severidade:** P1 — Critical
**Área:** SEO / robots.txt
**Data:** 2026-04-08
**Status:** ABERTO

### Comportamento Observado

O arquivo `https://smartlic.tech/robots.txt` contém dois blocos conflitantes para `Google-Extended`:

**Bloco 1 — Cloudflare Managed (linhas 1-80, aparece PRIMEIRO):**
```
User-agent: Google-Extended
Disallow: /
```

**Bloco 2 — SmartLic custom (aparece DEPOIS, mas não sobrepõe):**
```
# AI crawlers: Allow Google-Extended for AI Overviews citation eligibility.
# (apenas comentário — sem diretiva Allow explícita para Google-Extended)
```

Resultado: Google-Extended obedece ao primeiro bloco encontrado → `Disallow: /` → site inteiro bloqueado para AI Overviews.

### Comportamento Esperado

O comentário no código diz explicitamente "Allow Google-Extended for AI Overviews citation eligibility". O comportamento desejado é que Google-Extended possa rastrear o site. Para isso, a seção SmartLic custom precisa de:

```
User-agent: Google-Extended
Allow: /
```

### Notas Técnicas

- Verificado via: `curl -s https://smartlic.tech/robots.txt | grep -n "Google-Extended"`
  - Linha 50: `User-agent: Google-Extended` (Cloudflare block)
  - Linha 51: `Disallow: /`
  - Linha 62: apenas comentário na seção SmartLic
- Em robots.txt, quando o mesmo user-agent aparece em múltiplos blocos, o Googlebot segue as regras de todos os blocos aplicáveis (usa a mais restritiva, ou aplica a que vem primeiro — comportamento varia por crawler, mas Google-Extended não tem `Allow: /` explícito em nenhum lugar da seção SmartLic)
- Impacto: Sem elegibilidade para AI Overviews/SGE do Google — perde oportunidade de citação mencionada no playbook (83% AIO citations fora do top 10)

### Fix Sugerido

Em `public/robots.txt` (ou `next.config.js` se usando `headers`), na seção SmartLic custom, adicionar explicitamente:

```
# Google-Extended: permitido para elegibilidade em AI Overviews (Google SGE)
User-agent: Google-Extended
Allow: /
```

Isso deve aparecer APÓS o bloco Cloudflare managed, garantindo que o Google-Extended leia o Allow.

### ISSUE-SEO-002: Sitemap inclui URLs com query params (?modalidade=X) com canonical self-referencing

**Persona:** Marcus
**Severidade:** P1 — Critical
**Área:** SEO / Sitemap / Canonical
**Data:** 2026-04-08
**Status:** ABERTO

### Comportamento Observado

O sitemap inclui 5 versões de cada página setor×UF:

```
https://smartlic.tech/blog/licitacoes/vestuario/ac           ← canonical correto
https://smartlic.tech/blog/licitacoes/vestuario/ac?modalidade=6
https://smartlic.tech/blog/licitacoes/vestuario/ac?modalidade=4
https://smartlic.tech/blog/licitacoes/vestuario/ac?modalidade=8
https://smartlic.tech/blog/licitacoes/vestuario/ac?modalidade=12
```

**Pior ainda:** a tag canonical de cada URL parameterizada aponta para si mesma, NÃO para a URL limpa:
```html
<!-- Em /blog/licitacoes/vestuario/ac?modalidade=4 -->
<link rel="canonical" href="https://smartlic.tech/blog/licitacoes/vestuario/ac?modalidade=4"/>
```

### Impacto Mensurável

- 15 setores × 27 UFs × 5 variantes = **2.025 URLs** no sitemap
- Deveriam ser **405 URLs** canônicas (15 × 27)
- **1.620 URLs parasitas** desperdiçando crawl budget
- Google pode interpretar como 5 páginas de conteúdo duplicado por UF
- Ranking pode ser fragmentado entre variantes em vez de consolidado na URL canônica

### Comportamento Esperado

- Sitemap deve conter APENAS as 405 URLs canônicas (sem `?modalidade=X`)
- Se a página com `?modalidade=X` é acessível (200 OK), ela DEVE ter canonical apontando para a base: `https://smartlic.tech/blog/licitacoes/vestuario/ac`

### Notas Técnicas

- Verificado via:
  ```bash
  curl -s https://smartlic.tech/sitemap.xml | grep 'blog/licitacoes/vestuario/ac'
  # Retorna 5 URLs
  ```
  ```bash
  curl -s "https://smartlic.tech/blog/licitacoes/vestuario/ac?modalidade=4" | grep canonical
  # <link rel="canonical" href="https://smartlic.tech/blog/licitacoes/vestuario/ac?modalidade=4"/>
  ```
- Total URLs no sitemap: 3.923 (devia ser ~2.700 sem as variantes)

### Fix Sugerido

**Opção A (preferida):** Remover URLs com `?modalidade=X` do sitemap. Manter apenas a URL base. A filtragem por modalidade é um comportamento de UI — não precisa de URL indexável separada.

**Opção B:** Se páginas por modalidade têm conteúdo genuinamente diferente e valem indexar: corrigir o canonical para apontar para a URL base (`/blog/licitacoes/[setor]/[uf]`).

**Arquivo a editar:** Provavelmente `frontend/app/sitemap.ts` ou equivalente que gera o sitemap XML.

### ISSUE-SEO-003: CNPJ individual profiles não estão no sitemap (Onda 1 incompleta)

**Persona:** Jordan
**Severidade:** P2 — Important
**Área:** SEO / Sitemap / Onda 1
**Data:** 2026-04-08
**Status:** ABERTO

### Comportamento Observado

O commit `feat(seo): Onda 1 — CNPJ pages no sitemap (+5.000 URLs indexáveis)` prometeu adicionar páginas CNPJ ao sitemap.

Situação atual:
- `/cnpj` (landing page): **está no sitemap** ✅
- `/cnpj/33000167000101` (Petrobras): retorna **HTTP 200 + `robots: index, follow`** ✅
- No sitemap, porém, há **apenas 1 URL** (`/cnpj`) — nenhum CNPJ individual

### Impacto

- Páginas CNPJ são indexáveis (meta robots: index) mas o Googlebot não tem como descobri-las automaticamente sem serem linkadas ou estarem no sitemap
- O valor de SEO da Onda 1 está comprometido — as URLs existem mas não serão rastreadas ativamente

### Fix Sugerido

Duas opções:
1. **Gerar sitemap dinâmico de CNPJs**: Listar CNPJs mais relevantes (ex: top 1.000 por volume de contratos) e incluí-los no sitemap
2. **Internal linking**: Pelo menos linkar de páginas setor/UF para perfis CNPJ relevantes, permitindo rastreamento orgânico via links
